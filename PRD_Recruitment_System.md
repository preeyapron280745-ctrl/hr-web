## 1. บทสรุปผู้บริหาร (Executive Summary)

ระบบบริหารจัดการการสรรหาบุคลากร (Recruitment Management System) เป็นระบบเว็บแอปพลิเคชันภายในองค์กรที่ครอบคลุมกระบวนการสรรหาพนักงานตั้งแต่ผู้สมัครกรอกใบสมัครครั้งแรก จนผ่านการทดลองงาน สำหรับกลุ่มบริษัทในเครือ Comets Group (Comets Intertrade HQ, Comets Factory, ICT Manufacturing)

ระบบนี้จะมาทดแทนระบบเดิมที่พัฒนาด้วย AppSheet ซึ่งประสบปัญหาด้านประสิทธิภาพ ความเสถียร และการขยายขนาด (scalability) เมื่อปริมาณข้อมูลเพิ่มขึ้น โดยจะพัฒนาด้วย Google Apps Script (GAS) + HTML Service โดยใช้ Google Sheets เป็น backend

## 2. ปัญหาที่ต้องการแก้ไข (Problem Statement)

### 2.1 ปัญหาของระบบปัจจุบัน (AppSheet)

- **ประสิทธิภาพต่ำ**: ระบบช้าลงอย่างมีนัยสำคัญเมื่อข้อมูลสะสมเพิ่มขึ้น (ปัจจุบันมีใบสมัครในถังพัก 2,208 records และแนวโน้มเพิ่มขึ้นต่อเนื่อง)
- **ข้อมูลซับซ้อนเกินกว่า AppSheet จะรองรับได้ดี**: ฟอร์มใบสมัครมี 237 คอลัมน์ ทำให้โหลดช้าและ UI ไม่ตอบสนอง
- **Sheet จำนวนมาก**: ระบบมี 34 sheets ที่เชื่อมโยงกัน AppSheet ไม่สามารถจัดการ relationship ได้อย่างมีประสิทธิภาพ
- **ข้อจำกัดด้านการปรับแต่ง UI/UX**: ไม่สามารถออกแบบให้เหมาะสมกับ workflow เฉพาะทางของ HR ได้
- **ค่าใช้จ่าย License**: AppSheet มีข้อจำกัดในการใช้งานฟรีและค่า license เมื่อต้องการ feature ขั้นสูง

### 2.2 ผลกระทบต่อธุรกิจ

- HR เสียเวลาในการจัดการใบสมัครเนื่องจากระบบช้า
- หัวหน้าแผนกและผู้สัมภาษณ์ไม่สะดวกในการเข้าใช้งาน ทำให้การตอบกลับล่าช้า
- ข้อมูลประวัติผู้สมัครกระจัดกระจาย ไม่สามารถวิเคราะห์ภาพรวมของการสรรหาได้
- กระบวนการประเมินทดลองงาน 3 ครั้งไม่มีระบบติดตามที่ดี

## 3. เป้าหมายและตัวชี้วัดความสำเร็จ (Goals & Success Metrics)

### 3.1 เป้าหมาย (Goals)

1. **แทนที่ระบบ AppSheet** เดิมทั้งหมดภายในกรอบเวลาที่กำหนด โดยไม่กระทบกระบวนการสรรหาปัจจุบัน
2. **รองรับทั้ง 3 บริษัทในเครือ** (Comets HQ, Comets Factory, ICT) ด้วยระบบเดียว
3. **รองรับประเภทพนักงาน 3 แบบ**: รายเดือน, รายวัน, นักศึกษาฝึกงาน — ทั้ง 3 บริษัทต้องรองรับครบทุกประเภท รวมถึง ICT ด้วย
4. **ลดระยะเวลาเฉลี่ยในแต่ละขั้นตอนของ pipeline** เช่น ระยะเวลาที่หัวหน้าตอบกลับ, ระยะเวลาที่ผู้สัมภาษณ์บันทึกผล
5. **ปรับปรุง UX** ให้ใช้งานง่ายทั้งฝั่งผู้สมัคร (public form) และผู้ใช้ภายใน (HR, ผู้สัมภาษณ์, ผู้บริหาร)

### 3.2 ตัวชี้วัด (Success Metrics)

| ตัวชี้วัด | ค่าเป้าหมาย |
|---|---|
| เวลาโหลดหน้าใบสมัคร (first paint) | < 3 วินาที |
| เวลาโหลด HR Dashboard | < 5 วินาที (รองรับข้อมูล 5,000+ records) |
| อัตราการ submit ฟอร์มสำเร็จ | ≥ 98% |
| จำนวน error/bug ที่ร้ายแรง (P0/P1) หลัง go-live | < 5 เคส/เดือน |
| ความพึงพอใจของ HR และผู้สัมภาษณ์ | ≥ 4.0/5.0 |
| ระยะเวลาเฉลี่ยที่หัวหน้าตอบกลับ | ลดลง ≥ 30% เทียบกับระบบเดิม |

## 4. ขอบเขตของโครงการ (Scope)

### 4.1 In-Scope

- ฟอร์มสมัครงานสาธารณะ (Public Application Form) 12 ขั้นตอน
- HR Dashboard สำหรับจัดการ pipeline ทั้งหมด
- Interviewer Portal สำหรับหัวหน้าแผนก / ผู้สัมภาษณ์ (33 accounts)
- Admin / Master Data Management
- ระบบแจ้งเตือนผ่าน Email (อัตโนมัติ)
- ระบบ 3 รอบการประเมินทดลองงาน (25, 55, 85 วัน)
- การอัปโหลดไฟล์ (รูปถ่าย, Resume) ไปยัง Google Drive
- รองรับ 3 บริษัท × 3 ประเภทพนักงานครบทุก combination

### 4.2 Out-of-Scope (ในเฟสแรก)

- ระบบเงินเดือน / Payroll Integration
- ระบบ Performance Review สำหรับพนักงานประจำ (ไม่ใช่ทดลองงาน)
- Mobile Native App (ใช้ responsive web ก่อน)
- การเชื่อมต่อกับ Job Portal ภายนอก (JobsDB, JobThai)
- ระบบ Analytics ขั้นสูง (ใช้ Power BI ภายนอกเชื่อมต่อแทน)

### 4.3 Matrix การรองรับ บริษัท × ประเภทพนักงาน

| บริษัท | รายเดือน | รายวัน | นักศึกษาฝึกงาน |
|---|:---:|:---:|:---:|
| Comets Intertrade HQ (CM) | ✅ | ✅ | ✅ |
| Comets Factory | ✅ | ✅ | ✅ |
| ICT Manufacturing | ✅ | ✅ | ✅ |

ทุก combination ต้องรองรับได้ครบถ้วน โดยฟอร์มใบสมัครและฟิลด์ต่างๆ จะปรับตามประเภทพนักงานที่เลือก (เช่น นักศึกษาฝึกงานจะมี field เพิ่มเรื่องสถาบัน/อาจารย์ที่ปรึกษา)

## 5. กลุ่มผู้ใช้งาน (User Personas)

### 5.1 ผู้สมัครงาน (Applicant) – ภายนอก

- ต้องการกรอกใบสมัครผ่านมือถือหรือคอมพิวเตอร์
- ต้องการดูสถานะการสมัครของตนเอง
- ต้องการคอนเฟิร์ม/เลื่อนนัดสัมภาษณ์ผ่านอีเมล
- ต้องการยืนยันวันเริ่มงาน

### 5.2 เจ้าหน้าที่ฝ่ายบุคคล (HR Staff) – ภายใน

- ต้องการ**กรอก Form Resume เอง**สำหรับผู้สมัครที่เจอผ่านช่องทางนอก JobThai (Walk-in, Referral, LinkedIn, Job Fair) เพื่อเข้า pipeline โดยไม่ต้องรอผู้บริหารคัดเลือก
- ต้องการติดตามสถานะผู้สมัครในแต่ละขั้นของ pipeline
- ต้องการดู Resume ที่ผู้บริหารคัดเลือกมาจากถังพัก
- ต้องการส่งต่อ Resume ให้หัวหน้าแผนกพิจารณา (เลือกผู้พิจารณาสูงสุด 3 คน)
- ต้องการจัดการ Master Data (ตำแหน่งงาน, หัวหน้าแผนก, ผู้ใช้)

### 5.3 หัวหน้าแผนก / ผู้สัมภาษณ์ (Department Head / Interviewer) – ภายใน (33 accounts)

- ต้องการพิจารณาเรซูเม่ที่ HR ส่งมา (อนุมัติ/ไม่อนุมัติ/ส่งต่อแผนกอื่น)
- ต้องการบันทึกผลการสัมภาษณ์ตาม 14 หัวข้อการพิจารณา
- ต้องการประเมินผลการทดลองงาน 3 รอบ

### 5.4 ผู้บริหาร (Executive / CEO)

- ต้องการ**คัดเลือก Resume จาก "ถังพัก"** (ที่ดึงจาก JobThai) ผ่านอีเมล เพื่อกำหนดว่า Resume ใดควรเข้าสู่ pipeline
- ต้องการพิจารณาคำเสนอขอว่าจ้าง (อนุมัติ/ไม่อนุมัติ) หลังจากผ่านสัมภาษณ์
- ต้องการพิจารณาผลการทดลองงานครั้งสุดท้าย
- ต้องการดูภาพรวมการสรรหาผ่าน Dashboard

### 5.5 Admin ระบบ

- ต้องการจัดการสิทธิ์ผู้ใช้งาน
- ต้องการปรับปรุง Master Data
- ต้องการดู Log / Audit Trail

## 6. กระบวนการทำงาน (Pipeline Workflow)

### 6.1 Pipeline หลัก 8 ขั้นตอน

```
┌─────────────────────────────────────────────────────────┐
│ [1] Resume เข้าระบบ (2 ช่องทาง)                        │
│                                                          │
│   ช่องทาง A: ดึงจาก JobThai → "ถังพัก"                │
│   ช่องทาง B: HR กรอก "Form Resume" เองโดยตรง          │
└─────────────────────────────────────────────────────────┘
                         ↓
[2] ผู้บริหารคัดเลือกถังพัก (ผ่านอีเมล)  ◄── เฉพาะช่องทาง A
         ↓                                     ช่องทาง B ข้ามขั้นนี้
[3] Resume (ถังหลัก) → หัวหน้าแผนกพิจารณา (สูงสุด 3 คน)
         ↓
[4] หัวหน้าแผนกนัดวันสัมภาษณ์ (เสนอ 3 ช่วงเวลา)
         ↓
[5] ผู้สมัครกรอกใบสมัครฉบับเต็ม (237 cols) + คอนเฟิร์มวันสัมภาษณ์
         ↓
[6] ผู้สัมภาษณ์บันทึกผล (ใบสัมภาษณ์ + 14 หัวข้อพิจารณา)
         ↓
[7] ยืนยันวันเริ่มงาน + Onboarding
         ↓
[8] ทดลองงาน (ประเมิน 25 / 55 / 85 วัน)
```

**หลักการสำคัญของ flow นี้**:

1. **ถังพัก** คือ pool ของ Resume ที่**ดึงมาจาก JobThai** (import/scrape) มารอให้**ผู้บริหาร**คัดเลือกผ่านอีเมลว่าสนใจคนไหน เพื่อลดภาระ HR ในการคัดกรองจำนวนมาก
2. **Form Resume (bypass route)** — กรณี HR เจอผู้สมัครที่น่าสนใจผ่านช่องทางอื่น (เช่น Walk-in, Referral, LinkedIn, งาน Job Fair) HR สามารถกรอก Form Resume เพื่อเข้าถัง Resume ได้ทันที **โดยไม่ต้องรอผู้บริหาร**
3. **ใบสมัครฉบับเต็ม 237 cols** ผู้สมัครกรอกหลังถูกนัดสัมภาษณ์เท่านั้น — ลดภาระและเพิ่ม conversion rate

### 6.2 สถานะที่เป็นไปได้ในแต่ละขั้น

- **สถานะคัดเลือก**: รอพิจารณา / อนุมัติ / ไม่ผ่านการคัดเลือก
- **สถานะการคอนเฟิร์ม**: รอคอนเฟิร์ม / คอนเฟิร์มแล้ว / ปฏิเสธการสัมภาษณ์
- **สถานะการสัมภาษณ์**: รอสัมภาษณ์ / ผ่าน / ไม่ผ่าน / ไม่มาสัมภาษณ์
- **สถานะทดลองงาน**: รอประเมิน / ผ่าน / ไม่ผ่าน / ไม่มาเริ่มงาน
- **สถานะการพิจารณา (หัวหน้า)**: รอพิจารณา / อนุมัติ / ไม่อนุมัติ / ส่งต่อแผนกอื่น

## 7. ข้อกำหนดด้านการใช้งาน (Functional Requirements)

### 7.1 Module 1: Public Application Form (ใบสมัครฉบับเต็ม)

Module นี้เป็น**ฟอร์มสาธารณะที่ผู้สมัครกรอกหลังถูกนัดสัมภาษณ์** (ขั้นที่ 5 ของ pipeline) ไม่ใช่จุดเข้าระบบ เนื่องจาก Resume เข้าระบบผ่านช่องทาง JobThai import หรือ HR กรอก Form Resume (ดูข้อ 7.2)

ฟอร์มเป็นแบบ multi-step 12 ขั้นตอน (237 cols) เนื้อหาปรับตาม **ประเภทพนักงาน** ที่ผู้สมัครเลือก (รายเดือน / รายวัน / นักศึกษาฝึกงาน) ผู้สมัครจะเข้าถึงฟอร์มผ่าน**ลิงก์ที่ส่งมาในอีเมลเชิญสัมภาษณ์** พร้อมกับลิงก์คอนเฟิร์มช่วงเวลาสัมภาษณ์ 3 ช่วง

#### 7.1.1 ขั้นตอนในฟอร์ม

| ลำดับ | หัวข้อ | Fields หลัก |
|---|---|---|
| 1 | เลือกประเภทพนักงาน + บริษัท | ประเภทพนักงาน, บริษัท, ที่ตั้ง |
| 2 | ตำแหน่งที่ต้องการสมัคร | ตำแหน่ง (สูงสุด 3), ช่องทางที่ทราบข่าว, ผู้แนะนำ |
| 3 | เงินเดือน / ค่าตอบแทน | เงินเดือนปัจจุบัน, โอที, ค่ากะ, โบนัส, เงินเดือนที่ต้องการ, วันที่เริ่มงาน |
| 4 | ประวัติส่วนตัว | คำนำหน้า, ชื่อ (ไทย/อังกฤษ), เกิด, อายุ, สัญชาติ, เชื้อชาติ, ศาสนา, ส่วนสูง/น้ำหนัก |
| 5 | การติดต่อ | โทรศัพท์, อีเมล, Line ID, ที่อยู่, โรงพยาบาลประกันสังคม |
| 6 | สถานะครอบครัว | สถานะ, คู่สมรส, บิดา/มารดา, จำนวนพี่น้อง |
| 7 | ประวัติการศึกษา | ระดับการศึกษา, สถาบัน, สาขา, เกรดเฉลี่ย (7 ระดับ: ป.ถึงเอก) |
| 8 | หลักสูตรอบรม + ประสบการณ์ทำงาน | Sub-form: หลักสูตร, Sub-form: ประสบการณ์ทำงาน 3 บริษัทล่าสุด |
| 9 | ความสามารถ | ภาษา (ฟัง/พูด/อ่าน/เขียน), คอมพิวเตอร์ (Excel/Word/PPT/PS), ใบขับขี่, ทะเบียนรถ |
| 10 | ประวัติสุขภาพ | สูบบุหรี่/สุรา/ยาเสพติด, โรคประจำตัว, การผ่าตัด, การตั้งครรภ์, ประวัติจิตเวช |
| 11 | ข้อมูลติดต่อฉุกเฉิน + บุคคลอ้างอิง | ผู้ติดต่อฉุกเฉิน 2 คน, บุคคลอ้างอิงไม่ใช่ญาติ |
| 12 | อัปโหลดไฟล์ + ยืนยัน | รูปถ่ายหน้าตรง, Resume, ลงชื่อ, ยินยอม PDPA |

**สำหรับนักศึกษาฝึกงาน** เพิ่ม fields: ระยะเวลาฝึก, สถาบัน/คณะ/สาขา, อาจารย์ที่ปรึกษา, ลักษณะงานที่สนใจ

#### 7.1.2 Requirements

- **[F-1.1]** ฟอร์มต้อง auto-save ทุก step เพื่อไม่ให้ผู้สมัครต้องกรอกใหม่หากปิดหน้าก่อนเสร็จ
- **[F-1.2]** Validation ข้อมูล: เบอร์โทร, อีเมล, อายุ, ฟิลด์บังคับ
- **[F-1.3]** อัปโหลดไฟล์ไปยัง Google Drive ผ่าน OAuth scope (`https://www.googleapis.com/auth/drive`)
- **[F-1.4]** รองรับการใช้งานบนมือถือ (responsive)
- **[F-1.5]** แสดงข้อความยืนยันเมื่อ submit สำเร็จ พร้อม `รหัสใบสมัคร` ให้ผู้สมัครเก็บไว้อ้างอิง

### 7.3 Module 3: Interviewer Portal (หัวหน้าแผนก / ผู้สัมภาษณ์)

### 8.3 Relationship

- Primary Key: `ID` (ทุก sheet มี ID ของตนเอง)
- Foreign Key กลาง: `Key` / `รหัสใบสมัคร` (เชื่อมระหว่าง sheets ใน pipeline)
- **JobThai → ถังพัก**: Import อัตโนมัติ/กึ่งอัตโนมัติเข้า sheet "ถังพัก"
- **ถังพัก → Resume**: เมื่อ**ผู้บริหาร**อนุมัติผ่านอีเมล ระบบจะ copy/transform record เข้าสู่ "Resume"
- **Form Resume (HR) → Resume**: HR กรอก Form Resume โดยตรง ข้อมูลเข้า "Resume" ทันที (bypass ผู้บริหาร/ถังพัก)
- **Resume → วันนัดสัมภาษณ์**: เมื่อหัวหน้าแผนกอนุมัติเรซูเม่แล้วเสนอช่วงเวลาสัมภาษณ์ 3 ช่วง
- **วันนัดสัมภาษณ์ → ใบสมัคร**: เมื่อผู้สมัครได้รับลิงก์ จะกรอกฟอร์มเต็ม 237 cols พร้อมเลือกช่วงเวลาสัมภาษณ์
- **ใบสมัคร → ใบสัมภาษณ์ → ใบประเมิน**: ผ่านการเชื่อมด้วย `Key`

## 9. ข้อกำหนดด้านที่ไม่ใช่การใช้งาน (Non-Functional Requirements)

### 9.1 Performance

- **[NF-1.1]** หน้า HR Dashboard ต้องโหลดเสร็จภายใน 5 วินาทีที่ข้อมูล 5,000 records
- **[NF-1.2]** การ submit ฟอร์มใบสมัครต้องเสร็จภายใน 10 วินาที (รวมอัปโหลดไฟล์)
- **[NF-1.3]** ใช้ CacheService ของ GAS เพื่อลดการเรียก Sheets API ซ้ำ
- **[NF-1.4]** Lazy loading ของภาพและ section ที่ไม่จำเป็นต้องแสดงทันที

### 9.2 Security & Privacy

- **[NF-2.1]** Authentication ผ่าน Google Account + username/password (เก็บ hash)
- **[NF-2.2]** Role-based access control 4 ระดับ: Applicant / HR / Interviewer / Admin
- **[NF-2.3]** ผู้สัมภาษณ์เห็นเฉพาะผู้สมัครที่ได้รับมอบหมายเท่านั้น
- **[NF-2.4]** ข้อมูลที่ละเอียดอ่อน (บัตรประชาชน, สุขภาพ) ต้องมี audit log
- **[NF-2.5]** ปฏิบัติตาม PDPA: เก็บความยินยอมของผู้สมัคร + สิทธิขอลบข้อมูล

### 9.3 Scalability

- **[NF-3.1]** รองรับข้อมูลสะสม 50,000+ records ในระยะ 3 ปี
- **[NF-3.2]** รองรับ concurrent user 50+ คนโดยไม่ช้าลง
- **[NF-3.3]** Architecture สามารถย้ายจาก Google Sheets backend ไปยัง Database (เช่น Cloud SQL) ได้ในอนาคตโดยไม่ต้องเขียน frontend ใหม่

### 9.4 Usability

- **[NF-4.1]** UI ภาษาไทยทั้งหมด (column names, labels, messages)
- **[NF-4.2]** Responsive design (mobile / tablet / desktop)
- **[NF-4.3]** Error message ต้องอ่านเข้าใจง่าย ไม่ใช่ technical jargon
- **[NF-4.4]** Progress indicator ในฟอร์ม multi-step

### 9.5 Reliability

- **[NF-5.1]** Uptime ≥ 99% (ใน business hours)
- **[NF-5.2]** Auto-backup Google Sheet ทุกวัน
- **[NF-5.3]** Retry mechanism เมื่อ API call ล้มเหลว
- **[NF-5.4]** Graceful degradation: หาก Drive API ล่ม ยังคง submit form ได้ (แต่ queue ไฟล์ไว้)

## 11. บทเรียนจากการพัฒนาที่ผ่านมา (Key Learnings)

1. **AppSheet ไม่เหมาะกับข้อมูลขนาดใหญ่**: ระบบเดิมช้าอย่างรุนแรงเมื่อมีคอลัมน์มาก (237 cols) และ record หลักพัน ทำให้ต้องย้ายมา GAS
2. **GAS JavaScript มีข้อจำกัด**: syntax error (เช่น quote ไม่ escape ใน onclick) อาจทำให้ JS ทั้งหน้าพังเงียบๆ ใน iframe ของ Apps Script
3. **`addDynamicListeners()` ไม่เสถียร**: ต้องผูก state updates และ DOM logic เข้าไปในตัว click handler โดยตรง
4. **Google Sheets schema ต้อง precise**: column header อาจมี trailing space หรืออักขระพิเศษ – ต้อง match exact ในโค้ด GAS
5. **Drive upload ต้อง authorize ก่อน**: รัน `quickDriveCheck` ใน Script Editor ครั้งแรกเพื่อ grant permissions
6. **ใช้ `findSheet()` helper** ที่ทำ case-insensitive matching เพื่อป้องกัน error "sheet name mismatch"

## 13. Risk & Mitigation

| ความเสี่ยง | ผลกระทบ | แผนการป้องกัน |
|---|---|---|
| ปริมาณข้อมูลโตเร็วเกินกว่า Google Sheets รองรับ (10M cells limit) | สูง | แบ่ง sheet ตามปี + Archive ข้อมูลเก่า + แผนย้ายไป Database |
| GAS execution limit (6 นาที/execution) | กลาง | ใช้ batch operations + asynchronous processing ด้วย Time-based trigger |
| ผู้ใช้ไม่ยอมเปลี่ยนจาก AppSheet | สูง | Training + Parallel run 1 เดือน + รับฟังและปรับปรุง UX ตาม feedback |
| ข้อมูลสูญหายตอน migrate | สูง | Full backup ก่อน migrate + dry-run 2 รอบ + rollback plan |
| OAuth consent screen ต้องผ่าน verification | กลาง | เตรียมยื่น verification ล่วงหน้า หรือ deploy เป็น internal app ของ Workspace |
| ผู้สัมภาษณ์ 33 คนลืม password | ต่ำ | Self-service password reset ผ่านอีเมล |

### 15.1 รายการฟอร์ม Rejection พร้อมเหตุผล

| ฟอร์ม | เหตุการณ์ | ผู้กรอกเหตุผล |
|---|---|---|
| ไม่สนใจ Resume | HR คัดใน pre-screening | HR |
| ไม่ผ่านการคัดเลือก | หัวหน้าแผนกไม่อนุมัติ Resume | หัวหน้าแผนก |
| ปฏิเสธการสัมภาษณ์ | ผู้สมัครปฏิเสธไม่มาสัมภาษณ์ | HR (บันทึกจากผู้สมัคร) |
| ไม่ผ่านการสัมภาษณ์ | สัมภาษณ์แล้วไม่ผ่าน | ผู้สัมภาษณ์ / HR |
| ไม่มาเริ่มงานตามวันที่กำหนด | ผู้สมัครไม่มาเริ่มงาน (no-show) | HR |

### 15.2 Email Templates ที่ต้องมี

1. **แจ้งผู้บริหารให้คัดเลือก Resume จากถังพัก** (→ ผู้บริหาร) — สรุปชุด Resume ใหม่จาก JobThai พร้อมลิงก์ Approve/Reject
2. ขอให้หัวหน้าแผนกพิจารณา Resume (→ หัวหน้าแผนก หลังเข้าถัง Resume แล้ว)
3. แจ้งนัดสัมภาษณ์ 3 ช่วงเวลา + ลิงก์กรอกใบสมัครฉบับเต็ม + ลิงก์คอนเฟิร์มช่วงเวลา (→ ผู้สมัคร หลังหัวหน้าแผนกนัด)
4. ยืนยันวันสัมภาษณ์ที่ผู้สมัครเลือก (→ ผู้สมัคร + ผู้สัมภาษณ์ + HR)
5. แจ้งผลผ่านการสัมภาษณ์ + เอกสารที่ต้องเตรียม (→ ผู้สมัคร)
6. ขออนุมัติคำเสนอจ้าง (→ ผู้บริหาร)
7. แจ้งวันเริ่มงาน + ลิงก์ยืนยัน (→ ผู้สมัคร)
8. ครบกำหนดประเมินทดลองงาน (→ ผู้สัมภาษณ์ / หัวหน้าแผนก)
9. แจ้งผลการพิจารณา (→ ผู้สมัคร: ผ่าน / ไม่ผ่าน)
